# Authentication

CricPR API uses JWT (JSON Web Token) authentication to secure endpoints and manage user sessions. This guide covers everything you need to know about authentication, tokens, and security.

## Overview

The API uses a dual-token system:

<CardGroup cols={2}>
  <Card title="Access Token" icon="key">
    Short-lived token (30 minutes) used to authenticate API requests
  </Card>
  <Card title="Refresh Token" icon="rotate">
    Long-lived token (30 days) used to obtain new access tokens
  </Card>
</CardGroup>

## Token types

The API supports four token types:

| Token Type | Purpose | Expiration |
|------------|---------|------------|
| `access` | Authenticate API requests | 30 minutes |
| `refresh` | Obtain new access tokens | 30 days |
| `resetPassword` | Password reset flow | 10 minutes |
| `verifyEmail` | Email verification | 10 minutes |

<Note>
  Token expiration times are configurable via environment variables: `JWT_ACCESS_EXPIRATION_MINUTES`, `JWT_REFRESH_EXPIRATION_DAYS`, etc.
</Note>

## Authentication flow

<Steps>
  <Step title="Register or login">
    Obtain your initial access and refresh tokens
  </Step>
  <Step title="Include access token">
    Add the token to the Authorization header in all API requests
  </Step>
  <Step title="Refresh when expired">
    Use the refresh token to get a new access token when it expires
  </Step>
  <Step title="Logout">
    Invalidate tokens when done
  </Step>
</Steps>

## Register a new account

Create a new user account to get started:

<CodeGroup>
```bash cURL
curl -X POST https://api.cricpr.com/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{
    "name": "John Doe",
    "email": "john@example.com",
    "password": "password123",
    "role": "user"
  }'
```

```javascript JavaScript
const response = await fetch('https://api.cricpr.com/v1/auth/register', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    name: 'John Doe',
    email: 'john@example.com',
    password: 'password123',
    role: 'user'
  })
});

const { user, tokens } = await response.json();
```

```python Python
import requests

response = requests.post(
    'https://api.cricpr.com/v1/auth/register',
    json={
        'name': 'John Doe',
        'email': 'john@example.com',
        'password': 'password123',
        'role': 'user'
    }
)

data = response.json()
user = data['user']
tokens = data['tokens']
```
</CodeGroup>

### Request body

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | Yes | User's full name |
| `email` | string | Yes | Valid email address (must be unique) |
| `password` | string | Yes | Minimum 8 characters with at least one number and one letter |
| `role` | string | Yes | Must be `"user"` |
| `fcmToken` | string | No | Firebase Cloud Messaging token for push notifications |

## Login

Authenticate with existing credentials:

<CodeGroup>
```bash Email Login
curl -X POST https://api.cricpr.com/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{
    "email": "john@example.com",
    "password": "password123"
  }'
```

```bash Phone Login
curl -X POST https://api.cricpr.com/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{
    "phone": "+919876543210",
    "password": "password123"
  }'
```
</CodeGroup>

<Info>
  Users can login with either email or phone number. Provide only one identifier.
</Info>

### Request body

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `email` | string | Conditional | Email address (required if phone not provided) |
| `phone` | string | Conditional | Phone number (required if email not provided) |
| `password` | string | Yes | User password |
| `fcmToken` | string | No | Firebase Cloud Messaging token |

The response format is identical to registration.

## Using access tokens

Include the access token in the `Authorization` header for all authenticated requests:

<CodeGroup>
```bash cURL
curl -X GET https://api.cricpr.com/v1/matches \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
```

```javascript JavaScript
const response = await fetch('https://api.cricpr.com/v1/matches', {
  headers: {
    'Authorization': `Bearer ${accessToken}`
  }
});
```

```python Python
import requests

response = requests.get(
    'https://api.cricpr.com/v1/matches',
    headers={'Authorization': f'Bearer {access_token}'}
)
```
</CodeGroup>

### How it works

The API uses Passport.js with the JWT strategy:

1. The token is extracted from the `Authorization: Bearer <token>` header
2. The JWT is verified using the secret key configured in `JWT_SECRET`
3. The token payload is checked for `type: "access"`
4. The user is loaded from the database with populated roles and permissions
5. If valid, the user object is attached to `req.user`

<Warning>
  If authentication fails, you'll receive a `401 Unauthorized` response with the message "Please authenticate"
</Warning>

## Refresh tokens

When your access token expires (after 30 minutes), use the refresh token to obtain a new access token:

```bash cURL
curl -X POST https://api.cricpr.com/v1/auth/refresh-tokens \
  -H "Content-Type: application/json" \
  -d '{
    "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  }'
```

### Response

```json
{
  "access": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "expires": "2025-02-07T14:19:10.722Z"
  },
  "refresh": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "expires": "2025-03-09T13:49:10.722Z"
  }
}
```

<Tip>
  The refresh endpoint returns both a new access token AND a new refresh token. Always update both tokens in your application.
</Tip>

## Logout

Invalidate your refresh token when logging out:

```bash cURL
curl -X POST https://api.cricpr.com/v1/auth/logout \
  -H "Content-Type: application/json" \
  -d '{
    "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  }'
```

The endpoint returns `204 No Content` on success. After logout, the refresh token cannot be used to obtain new access tokens.

## Password reset flow

Reset a forgotten password using a secure token-based flow:

<Steps>
  <Step title="Request password reset">
    Send reset email with OTP code
    
    ```bash
    curl -X POST https://api.cricpr.com/v1/auth/forgot-password \
      -H "Content-Type: application/json" \
      -d '{"email": "john@example.com"}'
    ```
    
    Returns a hashed token:
    ```json
    {"token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."}
    ```
  </Step>
  
  <Step title="Verify OTP">
    User receives 6-digit OTP via email. Verify it:
    
    ```bash
    curl -X POST https://api.cricpr.com/v1/auth/verify-password \
      -H "Content-Type: application/json" \
      -d '{
        "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
        "otp": "123456"
      }'
    ```
    
    Returns a reset password token:
    ```json
    {"token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."}
    ```
  </Step>
  
  <Step title="Reset password">
    Use the reset token to set a new password:
    
    ```bash
    curl -X POST https://api.cricpr.com/v1/auth/reset-password?token=RESET_TOKEN \
      -H "Content-Type: application/json" \
      -d '{"password": "newPassword123"}'
    ```
    
    Returns `204 No Content` on success.
  </Step>
</Steps>

<Warning>
  Reset password tokens expire after 10 minutes. Users must complete the flow within this time window.
</Warning>

## Email verification

Verify user email addresses using a similar OTP flow:

<Steps>
  <Step title="Send verification email">
    ```bash
    curl -X POST https://api.cricpr.com/v1/auth/send-verification-email \
      -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
    ```
    
    Returns:
    ```json
    {"token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."}
    ```
  </Step>
  
  <Step title="Verify OTP">
    ```bash
    curl -X POST https://api.cricpr.com/v1/auth/verify-email \
      -H "Content-Type: application/json" \
      -d '{
        "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
        "otp": "123456"
      }'
    ```
    
    Updates `isEmailVerified` to `true` and returns the updated user object.
  </Step>
</Steps>

## OAuth authentication

CricPR supports Google and Apple Sign-In:

<CodeGroup>
```bash Google Sign-In
curl -X POST https://api.cricpr.com/v1/auth/google \
  -H "Content-Type: application/json" \
  -d '{
    "token": "GOOGLE_OAUTH_TOKEN",
    "fcmToken": "FIREBASE_TOKEN"
  }'
```

```bash Apple Sign-In
curl -X POST https://api.cricpr.com/v1/auth/apple \
  -H "Content-Type: application/json" \
  -d '{
    "token": "APPLE_IDENTITY_TOKEN",
    "fullName": "John Doe",
    "fcmToken": "FIREBASE_TOKEN"
  }'
```
</CodeGroup>

Both endpoints return the same response format as registration, creating a new account if the user doesn't exist or logging them in if they do.

## Security best practices

<CardGroup cols={2}>
  <Card title="Store tokens securely" icon="shield">
    Never store tokens in localStorage. Use secure httpOnly cookies or encrypted storage.
  </Card>
  <Card title="Use HTTPS" icon="lock">
    Always use HTTPS in production to prevent token interception.
  </Card>
  <Card title="Refresh proactively" icon="clock">
    Refresh access tokens before they expire to avoid interruptions.
  </Card>
  <Card title="Logout on inactivity" icon="user-slash">
    Implement automatic logout after periods of inactivity.
  </Card>
</CardGroup>

## Error responses

### 401 Unauthorized

Returned when authentication fails:

```json
{
  "code": 401,
  "message": "Please authenticate"
}
```

Common causes:
- Missing or invalid access token
- Expired access token (use refresh token)
- Token of wrong type (e.g., using refresh token as access token)

### 400 Bad Request

Returned for validation errors:

```json
{
  "code": 400,
  "message": "Email already taken"
}
```

### 404 Not Found

Returned when user doesn't exist:

```json
{
  "code": 404,
  "message": "No user found associated with this email"
}
```

## Admin authentication

For admin-only endpoints, use the admin login endpoint:

```bash cURL
curl -X POST https://api.cricpr.com/v1/auth/adminLogin \
  -H "Content-Type: application/json" \
  -d '{
    "email": "admin@example.com",
    "password": "adminPassword123"
  }'
```

<Warning>
  This endpoint verifies that the user has an admin role. Regular users will receive a `403 Forbidden` response.
</Warning>

## JWT payload structure

Access tokens contain the following payload:

```json
{
  "sub": "507f1f77bcf86cd799439011",
  "type": "access",
  "iat": 1738845652,
  "exp": 1738847452
}
```

| Field | Description |
|-------|-------------|
| `sub` | User ID (subject) |
| `type` | Token type (`access`, `refresh`, etc.) |
| `iat` | Issued at timestamp |
| `exp` | Expiration timestamp |

<Info>
  The JWT secret key is configured via the `JWT_SECRET` environment variable and should be kept secure.
</Info>
