## Overview

The CricPR API is built on a modern, scalable architecture designed for high-performance cricket match management. The system employs a clustered Node.js backend with distributed caching, real-time WebSocket communication, and robust data persistence.

## Request Flow

<Steps>
  <Step title="Client Request">
    Client sends HTTP/WebSocket request to the API
  </Step>

  <Step title="Load Balancing">
    PM2 distributes request to available worker using round-robin
  </Step>

  <Step title="Authentication">
    Passport middleware validates JWT token
    
    ```javascript
    const auth = passport.authenticate('jwt', { session: false });
    ```
  </Step>

  <Step title="Cache Check">
    Redis service checks for cached response
    
    ```javascript
    const cached = await redisService.get(cacheKey);
    if (cached) return cached;
    ```
  </Step>

  <Step title="Business Logic">
    Controller processes request and interacts with services
  </Step>

  <Step title="Database Query">
    Mongoose queries MongoDB with optimized indexes
  </Step>

  <Step title="Cache Update">
    Response is cached in Redis for future requests
    
    ```javascript
    await redisService.set(cacheKey, response, ttl);
    ```
  </Step>

  <Step title="Response">
    JSON response sent back to client
  </Step>
</Steps>

## Cluster Architecture

### Master-Worker Model

The application uses Node.js cluster module with PM2:

```javascript
if (cluster.isMaster) {
  // Master process
  - Fork workers based on CPU cores
  - Monitor worker health
  - Handle graceful shutdowns
  - Coordinate worker restarts
} else {
  // Worker process
  - Handle client requests
  - Connect to MongoDB/Redis
  - Process business logic
  - Report health to master
}
```

## Caching Strategy

### Cache Layers

```
┌─────────────────────────────────────┐
│         Application Memory          │ ← In-memory objects
├─────────────────────────────────────┤
│            Redis Cache              │ ← Shared cache
├─────────────────────────────────────┤
│          MongoDB Database           │ ← Persistent storage
└─────────────────────────────────────┘
```

### Cache Patterns

<CodeGroup>
```javascript Cache-Aside
// Check cache first
const cached = await redisService.get(key);
if (cached) return cached;

// Query database if cache miss
const data = await Model.find(query);

// Update cache
await redisService.set(key, data, ttl);
return data;
```

```javascript Write-Through
// Update database
const doc = await Model.create(data);

// Update cache
await redisService.set(key, doc, ttl);

return doc;
```

```javascript Cache Invalidation
// Delete from database
await Model.findByIdAndDelete(id);

// Invalidate cache
await redisService.del(`model:${id}`);
await redisService.delPattern('model:list:*');
```
</CodeGroup>

## Scalability Patterns

### Horizontal Scaling

```
Application Tier:
- Add more PM2 workers
- Distribute across multiple servers
- Use load balancer for distribution

Database Tier:
- MongoDB replica sets
- Read replicas for queries
- Sharding for large datasets

Cache Tier:
- Redis cluster mode
- Distributed cache across nodes
- Cache replication
```

### Vertical Scaling

```
- Increase server resources (CPU, RAM)
- Optimize Node.js heap size
- Tune MongoDB memory settings
- Increase Redis memory allocation
```
