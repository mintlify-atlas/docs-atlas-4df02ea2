## Overview

The CricPR API includes Docker configuration for containerized deployment across development, testing, and production environments.

## Docker Configuration

### Dockerfile

The application uses a lightweight Node.js Alpine image:

```dockerfile Dockerfile
FROM node:alpine

# Create app directory with proper permissions
RUN mkdir -p /usr/src/node-app && chown -R node:node /usr/src/node-app

WORKDIR /usr/src/node-app

COPY package.json ./

# Run as non-root user
USER node

RUN npm install

COPY --chown=node:node . .

EXPOSE 3000
```

<Note>
  The Dockerfile uses the `node` user for security, preventing the application from running as root.
</Note>

## Docker Compose Configurations

The project includes multiple Docker Compose files for different environments:

### Base Configuration

```yaml docker-compose.yml
services:
  node-app:
    build: .
    image: node-app
    env_file: .env.docker
    ports:
      - "3002:3002"
      - "3003:3003"
    depends_on:
      - mongodb
    networks:
      - node-network

  mongodb:
    image: mongo:4.2.1-bionic
    ports:
      - "27017:27017"
    volumes:
      - dbdata:/data/db
    networks:
      - node-network

volumes:
  dbdata:

networks:
  node-network:
    driver: bridge
```

## Deployment Steps

<Steps>
  <Step title="Prepare Environment Files">
    Create environment file for Docker:
    
    ```bash
    cp .env.example .env.docker
    ```
    
    Update MongoDB URL to use Docker service name:
    
    ```bash .env.docker
    MONGODB_URL=mongodb://mongodb:27017/cricpr
    ```
  </Step>

  <Step title="Build Images">
    Build the Docker images:
    
    ```bash
    docker-compose build
    ```
  </Step>

  <Step title="Start Services">
    Launch containers based on environment:
    
    <CodeGroup>
    ```bash Development
    npm run docker:dev
    # or
    docker-compose -f docker-compose.yml -f docker-compose.dev.yml up
    ```
    
    ```bash Production
    npm run docker:prod
    # or
    docker-compose -f docker-compose.yml -f docker-compose.prod.yml up
    ```
    
    ```bash Testing
    npm run docker:test
    # or
    docker-compose -f docker-compose.yml -f docker-compose.test.yml up
    ```
    </CodeGroup>
  </Step>

  <Step title="Verify Deployment">
    Check running containers:
    
    ```bash
    docker-compose ps
    ```
    
    View logs:
    
    ```bash
    docker-compose logs -f node-app
    ```
  </Step>
</Steps>

## Container Management

### Starting and Stopping

<CodeGroup>
```bash Start (Detached)
docker-compose up -d
```

```bash Stop
docker-compose down
```

```bash Restart
docker-compose restart
```

```bash Stop and Remove Volumes
docker-compose down -v
```
</CodeGroup>

### Updating Containers

Rebuild and update containers with latest changes:

```bash
npm run docker:dev-update
# or
docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d --build --remove-orphans
```

### Accessing Containers

<CodeGroup>
```bash Shell Access
docker-compose exec node-app sh
```

```bash Run Commands
docker-compose exec node-app npm run seed:all
```

```bash View Logs
docker-compose logs -f node-app
```
</CodeGroup>

## Volume Management

### Data Persistence

MongoDB data is persisted using Docker volumes:

```yaml
volumes:
  dbdata:
```

### Backup Database

```bash
docker-compose exec mongodb mongodump --out /backup
docker cp $(docker-compose ps -q mongodb):/backup ./backup
```

### Restore Database

```bash
docker cp ./backup $(docker-compose ps -q mongodb):/backup
docker-compose exec mongodb mongorestore /backup
```

## Networking

Services communicate through the `node-network` bridge network:

```yaml
networks:
  node-network:
    driver: bridge
```

### Service Discovery

Services can reference each other by service name:

- Application: `node-app:3002`
- MongoDB: `mongodb:27017`

## Production Best Practices

<AccordionGroup>
  <Accordion title="Security">
    - Don't expose unnecessary ports
    - Use secrets management for sensitive data
    - Run containers as non-root users
    - Keep base images updated
    - Scan images for vulnerabilities
  </Accordion>

  <Accordion title="Performance">
    - Use multi-stage builds to reduce image size
    - Set resource limits for containers
    - Use volume mounts carefully in production
    - Optimize layer caching
  </Accordion>

  <Accordion title="Monitoring">
    - Implement health checks
    - Monitor container resource usage
    - Centralize log collection
    - Set up alerts for container failures
  </Accordion>

  <Accordion title="Scaling">
    - Use Docker Swarm or Kubernetes for orchestration
    - Implement load balancing
    - Use external databases for production
    - Consider stateless application design
  </Accordion>
</AccordionGroup>

## Health Checks

Add health checks to your docker-compose.yml:

```yaml
services:
  node-app:
    healthcheck:
      test: ["CMD", "wget", "--quiet", "--tries=1", "--spider", "http://localhost:3002/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="Container Won't Start">
    Check logs for errors:
    
    ```bash
    docker-compose logs node-app
    ```
    
    Verify environment variables:
    
    ```bash
    docker-compose exec node-app env
    ```
  </Accordion>

  <Accordion title="Database Connection Issues">
    Ensure MongoDB container is running:
    
    ```bash
    docker-compose ps mongodb
    ```
    
    Check network connectivity:
    
    ```bash
    docker-compose exec node-app ping mongodb
    ```
  </Accordion>

  <Accordion title="Port Conflicts">
    Change exposed ports in docker-compose.yml:
    
    ```yaml
    ports:
      - "3005:3002"  # host:container
    ```
  </Accordion>

  <Accordion title="Volume Permission Issues">
    Fix permissions on mounted volumes:
    
    ```bash
    sudo chown -R 1000:1000 ./data
    ```
  </Accordion>
</AccordionGroup>
